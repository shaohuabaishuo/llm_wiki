Step 1: 单模型 + Vanilla 27B 内存优化

Unpin Coder-14B → 只用 Qwen3.6-27B-4bit → chunked_prefill: true

关键发现: 内存压力完全解除，但 Token 速度几乎不变。说明瓶颈不在内存，而在别处。

Step 2: 消灭 Thinking Tokens 8.6x 加速

Dashboard → Models → Qwen3.6-27B-4bit → 关闭 Enable Thinking

洞察: Qwen3.6-27B-4bit（非 MTPLX 版本）也在 Claude Code system prompt 触发下输出思维链。有效 token 占 35-50%。关闭后有效吞吐翻倍。

Step 3: GDN Cache 问题 → 切换模型 架构决策

Qwen3.6-27B → Qwen2.5-Coder-14B：标准注意力避开 GDN cache 兼容性问题

01:06:38  Claude Code 请求进入
01:07:27  Prefix cache 被拒绝 → "rejecting cache to prevent stale GDN state"
01:11:22  第一个工具调用完成 (4分钟后!)
01:12:35  再次 cache 拒绝
01:14:28  请求被中止 (8分钟，未正常完成)
GDN (GatedDeltaNet) 是有状态注意力机制。oMLX SSD prefix cache 匹配到不完整块时，为保证状态一致性直接拒绝缓存 → 强制从头 prefill。标准注意力的 Qwen2.5 无此问题。

关键技术洞察

1. Token 速度 ≠ 体感速度

14.9 tok/s 指标不差，但思维链吃 50% 预算后 有效吞吐仅 7-8 tok/s。Dashboard 数字不能直接信。

2. GDN 是 Agent 场景的致命弱点

GDN 在单轮推理更高效，但在多轮 Claude Code 中 SSD prefix cache 被频繁拒绝 → 每次对话重新 prefill。标准注意力的 14B 虽然参数少，cache 稳定 → 总体更快。

3. 小模型 + 大 Context > 大模型 + 小 Context

14B+96K (56K 工作空间) 体验远超 27B+64K (24K 空间)。生成质量够用 + 不打断工作流 > 高质量但频繁 compact。

4. Metal Cap = 37.4 GB 不可逾越

48GB Mac 可用预算仅 ~20-25 GB (模型 + KV cache)。任何组合超过 37.4 GB 必崩。14B+96K=22GB 安全，14B+131K=37GB 撞墙。
